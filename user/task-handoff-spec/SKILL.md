# Task handoff spec (scout conventions → spec with rationale)

Battle-tested playbook (a production backend,
2026-06: an iOS-session-authored spec for a backend endpoint, executed later by a separate
backend session with zero clarifying questions). The core insight: the executor has none
of the context that made your decisions obvious — a spec is good exactly to the degree it
survives that loss.

## Why this shape

- **Scout before you spec.** The target repo already has patterns to reuse — error enums,
  helper functions, FK cascade rules, router/test conventions. A spec written without
  discovering them forces the executor to invent what exists (or worse, duplicate it).
  One read-only Explore agent over the target repo answers this in minutes.
- **Decisions carry their "why" or they get reverted.** A bare instruction that looks
  over-engineered ("delete the linked UserSpread too") will be "simplified" back by an
  executor without the discussion context. The same instruction with its rationale
  ("POST /readings is idempotent by user_spread_id — if the spread survives, a stale
  retry silently REGENERATES the deleted reading, a paid LLM call") is safe.
- **file:line citations make the spec re-verifiable.** Specs go stale; an executor who
  can jump to the cited line can tell instantly whether the claim still holds.

## The recipe

1. **Locate the task convention.** Find the target repo's task directory and numbering
   (`tasks/NNNN-slug/plan.md`-style), the next free number, and an existing task to
   copy the shape from. Match its language and section order — the executor's muscle
   memory is part of the interface.
2. **Scout the target repo** (read-only; an Explore agent fits). Collect: how the nearest
   existing feature does it (router/handler/component patterns), reusable helpers and
   enums, data-model constraints (FK cascades, unique indexes), test fixtures and
   conventions, the default branch and merge flow. Never touch the target repo's code —
   the spec is a doc-only contribution.
3. **Write the spec** with these sections (rename to taste, keep the roles):
   - **Header**: branch name to use, base branch @ SHA, scope boundary ("backend only —
     client follows in <ref>").
   - **Why**: the pain, in one paragraph, with the cross-repo context the executor lacks.
   - **Surface**: the endpoint/API/UI being added — methods, status codes, error codes
     (reuse existing enums; say so explicitly).
   - **Conventions discovered (reuse)**: the scout results, each with file:line.
   - **Behaviour / decisions**: every non-obvious choice WITH rationale and the edge it
     guards. This section is where the handoff lives or dies.
   - **Files**: the expected touch list (new and modified).
   - **Tests**: concrete cases, not "add tests" — happy path, repeat/idempotency,
     foreign-owner, legacy rows, and the adversarial case that pins the rationale
     (e.g. "after delete, replay with the dead id → 404, no silent regeneration").
   - **Acceptance**: the checklist that means done, including lint/test commands.
   - **Follow-up (out of scope)**: what this unblocks on the other side, with a pointer
     to where that work is tracked.
4. **Date it and cite it.** Put the authoring date in the header; every factual claim
   about the target repo gets a file:line. The executor re-verifies cheaply; a stale
   spec fails loudly instead of silently.

## Gotchas

- **Don't spec from memory of the target repo** — even your own. The scout phase exists
  because repos move; in the source run, half the "obvious" facts (which FK cascades,
  which error codes already exist) needed checking and one was wrong.
- **The adversarial test case is the rationale's enforcement.** If a decision guards an
  edge (idempotent replay, race, legacy rows), spec the test that fails when the
  executor "simplifies" the decision away. Prose alone doesn't survive contact.
- **Mind upstream/priority docs.** If the target repo mirrors a contract from another
  source (e.g. `docs/flow.md` generated by a backend release), say which copy is
  authoritative and when to re-sync — otherwise the executor fixes wording the upstream
  is about to overwrite.
- **State the scope boundary explicitly.** A spec authored from repo A for repo B should
  say what repo A will do afterwards (client adoption, fixture updates) so the executor
  doesn't helpfully reach across the boundary.
